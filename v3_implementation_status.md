# Bina v3 Upgrade — Implementation Status

This document summarizes the work completed during the Bina v3 architecture upgrade, outlines what was intentionally changed, and details the final missing pieces that remain to be implemented.

---

## 🟢 Phase 1: Completed Backend Core Changes

1. **Per-Workspace Configuration (`config.py` & `store.py`)**
   - Added support for multiple processing paths (`hosted`, `local`, `user_api`).
   - Extended the SQLite `workspaces` table to store:
     - `processing_path`, `model_name`, `embed_model`
     - `user_api_key`, `user_api_base`
     - `vector_backend` (defaults to `moorcheh`)
   - Implemented an `_utcnow()` helper across the app to resolve timezone deprecation warnings.
   - Built a dynamic migration in `store.py` to seamlessly add new columns to existing databases.

2. **Unified Inference Abstraction (`inference.py`)**
   - Created `inference.py` to handle all LLM calls (`call_inference`) and embedding generation (`embed_text`).
   - Supports routing to OpenAI-compatible hosted paths (HuggingFace/GPT-OSS), local Ollama (`qwen3.5:2b`), and custom user keys.
   - Extracts and standardizes JSON responses, stripping out `<think>...</think>` tags generated by reasoning models.

3. **Pluggable Vector Store (`vector_store_router.py`)**
   - Implemented `vector_store_moorcheh.py` using the `moorcheh-sdk` wrapper for the Moorcheh API.
   - Renamed existing ChromaDB implementation to `vector_store_local.py`.
   - Created `vector_store_router.py` to dynamically serve the correct vector store wrapper based on the workspace setting.
   - Left a shim `vector_store.py` so legacy imports continue to work gracefully.

4. **Pipeline and Search Rewrites (`pipeline.py`, `search.py`, `graph.py`)**
   - `pipeline.py`: Rewritten to use `inference.py` instead of raw `ollama` calls. Now accurately reflects workspace settings (models + backends).
   - `search.py`: Query embeddings now go through `inference.embed_text`. Search strictly queries the appropriate vector store router.
   - `graph.py`: Fixed all direct vector store lookups to use the dynamic router.

---

## 🟢 Phase 2: Completed Capabilities & API Endpoints

1. **Railtracks Reasoning Agent (`agent.py`)**
   - Created `agent.py` setting up `railtracks`. 
   - Wrapped Bina's core operations into agent tools: `semantic_search`, `answer_query`, `summarize_node`, `get_node_neighbors`.
   - Falls back gracefully to standard "summary concatenation" if `railtracks` fails or is unavailable.

2. **V3 API Upgrades (`api.py`)**
   - Replaced all raw ChromaDB lookups (e.g. vector counting) with the dynamic router.
   - Added `POST /query` endpoint combining direct semantic searching (`mode="search"`) and Railtracks-powered reasoning (`mode="agent"`).
   - Added `GET /workspaces/:id/model` and `PATCH /workspaces/:id/model` endpoints to allow updating a workspace's AI settings on the fly.
   - Allowed defining the configuration variables via `POST /workspaces` payload.

---

## 🟢 Phase 3: Completed Frontend Work

1. **Types & API Layer (`types.ts`, `api.ts`)**
   - Added new selection fields to the `Workspace` interface.
   - Added the new `QueryResult` typing for agent chats.
   - Added implementations for `workspacesApi.getModel()` and `query()`.

2. **Conversational "Ask Bina" UI (`AskBinaPanel.tsx`, `MainLayout.tsx`)**
   - Created `AskBinaPanel.tsx` delivering a ChatGPT-style conversational interface.
   - Supports cited sources attached directly to the answers.
   - Created a top navigation toggle in `MainLayout.tsx` letting users seamlessly swap between the **Graph view** and the **Ask view**.

3. **Smart Model Choice in Onboarding (`Onboarding.tsx`)**
   - Injected a `model_choice` step before folder selection.
   - Users can now select "Hosted AI", "Local AI", or "Your API Key".
   - Only boots into the Ollama model downloader screen if "Local AI" is specifically chosen. 
   - Dynamically requires the API Key text input if "Your API Key" is selected.

## 🔴 Missing Tasks & Next Steps (To-Do)

While the heavy lifting and end-to-end framework are fully implemented, a few pieces need tying up to finalize v3.

### 1. Workspace Settings UI (Post-Onboarding)
- **What's built:** The backend `PATCH /workspaces/:id/model` endpoint is ready, and users can set their model path *during onboarding*.
- **Task:** We need a UI settings panel inside the app (e.g., in `WorkspaceModal.tsx` or a new Sidebar modal) where the user can update an *existing* workspace's processing path. For example, migrating an older workspace from "Local" to "Hosted".

### 2. Global Moorcheh API Key Entry
- **What's built:** Moorcheh reads `MOORCHEH_API_KEY` from `~/.bina/.env`.
- **Task:** There is currently no UI in the frontend settings screen to let the user paste an API key if one doesn't already exist in the environment files. A global app settings screen section needs to be built for this.

### 3. Graceful Error Handling Feedback
- **What's built:** `AskBinaPanel.tsx` blindly reports error messages if backend fails.
- **Task:** If Moorcheh is missing an API Key or throws a 401 Unauthorized, the UI should gracefully intercept it and show a user-friendly configuration prompt instead of a raw traceback string.

### 4. Cleanup of Pyre Linter Import Warnings
- **What's built:** The code runs perfectly.
- **Task:** Pyre2 (the Python linter) complains about imports in `/backend/api.py` not resolving because `sys.path.insert(0, ...)` is done at runtime. To quiet warnings, you'd likely want to configure the IDE paths or move `api.py` directly into the root `Bina2/` directory rather than the `/backend/` directory.

### 5. Final e2e Test Walkthrough
- **Task:** Build/run the Electron renderer and visually confirm that the newly created Ask panel styles look exactly as intended with Tailwind classes, ensuring no React warnings appear in the Developer Console.
